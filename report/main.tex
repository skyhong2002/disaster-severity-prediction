velda.cs14@nycu.edu.tw), and Sky Shih-Kai Hong (314554024,

\href{https://github.com/skyhong2002/disaster-severity-prediction}{\color{blue}https://github.com/skyhong2002/disaster-severity-prediction}.

Natural disaster severity prediction is a critical task for risk management
and early warning systems. In this work, we address the problem of forecasting
weekly drought severity scores (0--5 scale) for 2,248 geographic regions using
historical meteorological data. The current implementation is a two-stage
LightGBM pipeline: a score reconstructor first estimates missing score-history
signals from meteorological features, and five direct horizon models then
forecast the next five weekly scores. The implemented system builds 327
features per horizon from calendar variables, meteorological lags, rolling
statistics, exponentially weighted means, region-level statistics, and score
history. To support future iterations, every training run is stored as a
versioned experiment containing its configuration, metrics, model artifacts,
and submission metadata. On the local holdout split, the five horizon models
obtain an average MAE of \textbf{0.2198}. The submitted Kaggle public
leaderboard score is \textbf{0.8094}, evaluated on approximately 40\% of the
test data.
\end{abstract}

\section{Project Summary (\textcolor{red}{max 1 page})}

\subsection{Goal}
The goal of this project is to predict the drought severity score for each
geographic region over the \textbf{next five consecutive weeks} (35 days),
given 91 days of historical meteorological observations per region from the
test set. Scores range from 0 (no disaster) to 5 (extreme severity).

\subsection{High-Level Approach}
Our current baseline follows a supervised machine learning paradigm treating
the task as a \textbf{multi-step time series regression} problem. We extract
temporal features from the raw meteorological columns, including lag values,
rolling means/stds, exponentially weighted averages, calendar encodings,
historical score summaries, and per-region score statistics. These features
feed into one LightGBM model per prediction horizon. Final predictions are
post-processed by clipping values to the valid score interval [0, 5]. The
report is structured so that future algorithms can be added as new experiment
versions rather than replacing the current LightGBM baseline.

\subsection{Related Works}
Time series forecasting for environmental phenomena has been widely studied.
Gradient boosting methods, notably XGBoost~\cite{chen2016xgboost} and
LightGBM~\cite{ke2017lightgbm}, have demonstrated strong performance on
tabular time series data by capturing non-linear feature interactions without
requiring extensive feature scaling. For forecasting competitions and
meteorological applications, lag-window feature engineering is often effective
because recent observations and seasonal patterns usually contain strong
predictive signals~\cite{kaggle_m5}. Transformer-based models such as Temporal
Fusion Transformer~\cite{lim2021temporal} and PatchTST~\cite{nie2023patchtst}
are possible future extensions, but the current repository implementation
focuses on a fully reproducible gradient-boosted tree pipeline.

\section{Proposed Method (\textcolor{red}{1-3 pages})}

\subsection{Problem Formulation}
Let $\mathcal{R} = \{r_1, r_2, \ldots, r_K\}$ denote the set of geographic
regions. For each region $r_k$, we observe a time series of meteorological
features $\mathbf{X}_{r_k} \in \mathbb{R}^{T \times D}$ where $T$ is the
number of observed days and $D$ is the number of meteorological variables.
The target $y_{r_k, t}$ is the drought severity score at week $t$, available
only on a specific day per week (the rest are NaN).

Given the 91-day test window, the task is to predict:
\begin{equation}
  \hat{y}_{r_k, t+1}, \hat{y}_{r_k, t+2}, \ldots, \hat{y}_{r_k, t+5}
\end{equation}
\noindent minimizing the Mean Absolute Error:
\begin{equation}
  \text{MAE} = \frac{1}{K \cdot 5} \sum_{k=1}^{K} \sum_{w=1}^{5}
  \left| \hat{y}_{r_k, w} - y_{r_k, w} \right|
\end{equation}

\subsection{Data Preprocessing}

\subsubsection{Handling NaN Scores}
Scores are observed only once per week, so most daily rows have missing target
values. During training, only rows with real non-missing scores are used as
supervised labels. For score-history features, the observed weekly scores are
forward-filled within each region and then shifted before feature construction,
so that every score-history variable uses only past information. For the test
window, real scores are unavailable; therefore, a separate Stage-1 model
reconstructs plausible weekly scores from meteorological and calendar features
before the Stage-2 forecasting models are applied.

\subsubsection{Feature Engineering}
For each meteorological column $d$, we construct the following feature groups:

\begin{itemize}
  \item \textbf{Meteorological lags}: 7, 14, 21, 28, 35, 42, and 49 days.
  \item \textbf{Rolling statistics}: shifted rolling mean and standard
        deviation over 7, 14, 21, 28, and 35 days.
  \item \textbf{EWM features}: exponentially weighted means with spans 3, 7,
        and 14 days.
  \item \textbf{Calendar features}: month, quarter, day of month, approximate
        day of year, week of year, and cyclical sine/cosine encodings.
  \item \textbf{Score history}: lagged weekly scores at 1, 2, 3, 4, 5, 6, 8,
        10, and 12 weeks, plus rolling score mean/std over 4, 8, 12, 26, and
        52 weeks.
  \item \textbf{Region statistics}: historical mean, standard deviation,
        median, maximum, and minimum score per region, computed from training
        data only.
\end{itemize}

The final Stage-2 feature matrix contains 327 model input columns.

\subsection{Model Architecture}

\subsubsection{Stage-1 Score Reconstructor}
The first model predicts the observed weekly score using meteorological,
calendar, and region-level features, excluding all score-lag and score-rolling
features. This model is used only during inference to fill the score signal
inside the 91-day test window. It uses LightGBM with an MAE objective and 1,000
estimators, and obtains a local validation MAE of 0.4869.

\subsubsection{Stage-2 Direct Horizon Models}
We train one LightGBM regressor per prediction horizon (week 1 through week 5),
a technique known as \textbf{Direct Multi-Step Forecasting}. Each horizon model
uses the full feature set and predicts exactly one target column,
\texttt{target\_w1} through \texttt{target\_w5}. The main hyperparameters are:
\begin{itemize}
  \item \texttt{num\_leaves}: 127
  \item \texttt{learning\_rate}: 0.05
  \item \texttt{n\_estimators}: 3000 with early stopping
  \item \texttt{feature\_fraction}: 0.8
  \item \texttt{bagging\_fraction}: 0.8, \texttt{bagging\_freq}: 1
  \item \texttt{objective}: \texttt{regression\_l1}, \texttt{metric}: MAE
\end{itemize}

\subsection{Inference and Submission Generation}
At inference time, the raw training and test rows are concatenated and sorted
by \texttt{region\_id} and \texttt{date}. Stage 1 reconstructs score values for
the test window. Stage 2 rebuilds the full feature table using these
reconstructed scores, selects the last row of the 91-day test window for each
region, and applies the five horizon models. The prediction script outputs a
timestamped CSV file under \texttt{submissions/}, containing \texttt{region\_id}
and five weekly prediction columns.

\subsection{Experiment Versioning}
Because the implementation is expected to evolve, each run is saved under a
timestamped directory \texttt{experiments/<run\_id>/}. The directory contains
\texttt{config.json} for hyperparameters and feature settings,
\texttt{metrics.json} for validation results, and
\texttt{submission\_metadata.json} for prediction statistics after inference.
Large model files and generated submission CSVs are stored inside the same run
directory but are ignored by git. This format allows future algorithms such as
XGBoost, additional LightGBM variants, ensembles, or neural time series models
to be appended to the report as comparable experiment rows.

The current implementation uses a chronological holdout split: after merging
features with weekly labels, the last 20\% of samples are used as validation
data and the earlier 80\% are used for training. This split preserves temporal
ordering and avoids random shuffling across future observations. Model files
are saved as \texttt{models/lgbm\_score\_reconstructor.pkl} and
\texttt{models/lgbm\_models.pkl}.

\section{Conclusion}
In this work, we implemented a complete time series prediction framework for
forecasting drought severity levels from meteorological data. The current
system contains data downloading, memory reduction, temporal feature
engineering, two-stage LightGBM training, model serialization, and Kaggle
submission generation. Local validation results show that direct horizon
models become less accurate as the forecast horizon increases, from 0.1412 MAE
at week 1 to 0.2861 MAE at week 5. On the Kaggle public leaderboard, Team 5
achieved 0.8094 MAE with the first submission, ranking close to Baseline 3 and
above Baseline 2 and Baseline 1 in the provided snapshot. Future work includes
running formal ablation experiments, improving the validation strategy, tuning
feature groups and hyperparameters, and comparing the current LightGBM
pipeline with XGBoost or Transformer-based time series models.
