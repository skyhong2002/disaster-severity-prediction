## 核心概要

期末專案的真實重點不是「做一個 Kaggle 模型」，而是把團隊協作、時間序列預測、可重現報告與學術誠信綁成同一個評分系統；課程團隊明確把競賽表現、報告品質、進度檢查與提交一致性全部納入成績，因此任何只衝排行榜、忽略報告與程式可驗證性的策略都是短視。專案要求各組使用 NYU／NYCU 信箱登入平台、建立隊伍、以每個區域過去 91 天的特徵預測未來 5 週災害嚴重度，評估指標為 MAE，公開榜與私人榜分開計分，且每日每隊僅有 3 次提交機會；這意味著模型迭代必須有紀律，不能靠盲目刷榜。報告端同樣不是形式作業：課程團隊要求 IEEE 格式、英文撰寫、5 至 8 頁正文、附 GitHub 公開連結與 README，並且報告內容必須與程式、結果一致；助教也直接警告抄襲、程式不一致、AI 生成未查證引用與幻覺內容都可能導致低分或零分。真正的槓桿在於把資料建模、實驗記錄與報告敘事從第一天就整合起來；我們若把這件事當成單純模型競賽，會輸在可解釋性與評分合規，若把它當成完整研究交付，才有機會同時拿到競賽分、報告分與進度分。

## 期末專案的評分戰場：模型、報告、進度三線作戰

### 1. 競賽本體：用 91 天資料預測未來 5 週災害嚴重度

期末專案是一個團隊制 Kaggle 類競賽，佔學期成績 35%。任務是根據歷史氣象／災害相關特徵，預測各區域未來 5 週的自然災害嚴重度分數。

核心資料結構如下：

- **資料檔案**：`train.csv`、`test.csv`、`sample_submission.csv`
- **主要欄位**：
  - `region_id`：區域唯一識別碼。
  - 日期／時間相關欄位：資料以日為粒度呈現。
  - 多個特徵欄位：作為模型輸入。
  - `score`：訓練資料中的目標標籤，範圍為 0 到 5，數值越高代表災害嚴重度越高。
- **標籤特性**：`score` 是週級資料，因此一週中只有某一天有值，其餘可能為 `N/A`。
- **測試資料限制**：每個區域只提供 91 天特徵，沒有 `score` 欄位。
- **預測輸出**：每個 `region_id` 需輸出未來 5 週的預測值，即 `week1` 到 `week5`。

這裡的建模陷阱很明確：訓練時不必固定只用 91 天視窗，但最終模型必須能在測試階段只根據 91 天特徵產生未來 5 週預測。助教也指出一種可能策略：先用 91 天特徵推估對應的 `score` 序列，再用該序列輔助預測未來 5 週分數。這不是唯一答案，但它揭示了評分任務的本質：模型需要處理「日級特徵」與「週級標籤」之間的不對齊問題。

### 2. 排行榜與提交規則：不能靠暴力試錯

競賽評估使用 **MAE**。平台設有公開排行榜與私人排行榜，最終系統會從公開榜表現較好的提交中選取結果進入私人榜評估，規則與 Assignment 3 類似。

提交限制與評分權重如下：

- **每日提交限制**：每隊每天 3 次。
- **提交單位**：以團隊為單位，不是個人。
- **競賽分數組成**：
  - 公開榜分數佔競賽部分的 30%。
  - 私人榜分數佔競賽部分的 70%。
- **專案總體評分提及**：
  - Kaggle／競賽表現與報告共同構成主要成績。
  - 報告佔重要比例。
  - 進度檢查另有 5 分。

這套設計的潛台詞很硬：公開榜只是局部訊號，私人榜才是主戰場。團隊如果只對公開榜過度擬合，可能在最終私人榜失血。正確打法應該是建立本地驗證集、記錄每次提交假設，並限制「無原因提交」。

### 3. 團隊與平台管理：帳號錯誤會變成低級風險

課程團隊反覆要求學生使用 NYU／NYCU 信箱登入平台，不要使用 Gmail。已有學生用 Gmail 登入作業，教師要求更換信箱。

團隊操作要求包括：

- 登入平台後進入 Team 頁面。
- 透過隊友的使用者名稱送出邀請。
- 右側較小字樣為平台 username，邀請時需確認使用該名稱。
- 若隊伍名稱受限制，可在平台中修改名稱。
- 每隊每日提交上限為 3 次。

這部分不是技術難點，但它是典型的行政地雷。帳號、隊伍、提交者若不一致，後續成績對應與提交認定都可能出問題。

### 4. 報告要求：不是附錄，而是另一個主戰場

報告必須以論文格式撰寫，課程提供 IEEE Overleaf 模板，也允許使用 Microsoft Word 的 IEEE 模板。正文頁數限制為 **5 到 8 頁**，不含參考文獻；參考文獻頁數不限。報告必須以英文撰寫。

必備結構如下：

- **摘要**
  - 包含 group ID。
  - 包含 GitHub 公開連結。
  - GitHub repo 必須包含 README，說明如何執行程式。
- **Paper Summary**
  - 最多 1 頁。
  - 說明專案目標、方法高層想法、相關研究或自然災害預測相關工作。
- **Method**
  - 1 到 3 頁。
  - 說明模型設計。
  - 建議加入公式、偽程式碼、架構圖。
- **Experiment**
  - 2 到 4 頁。
  - 描述資料、統計分析、實驗設定。
  - 必須包含目前模型效能表格。
  - 可比較不同提交版本、不同模型變更帶來的效果。
  - 鼓勵加入自定義 baseline、ablation study、parameter analysis、case study。

格式要求也很明確：不得修改模板的基本格式，包括字體大小與行距。若被發現更改格式以塞內容，報告分數可能被扣。

這裡的戰略重點是：報告不是賽後補寫，而是實驗紀錄的外顯化。沒有實驗版本紀錄、沒有 baseline、沒有可重現 README，最後很難寫出有說服力的 5 到 8 頁。

### 5. 進度檢查與選擇性口頭報告：分數之外的曝光機會

進度檢查安排在 **5 月 21 日**，佔 5 分。每組需準備約 5 分鐘簡報，說明目前進度，例如資料分析觀察、目前挑戰、方法高層想法等。助教明確表示，不需要揭露完整演算法細節；若擔心其他組拿走想法，可以只分享高層設計。但分享內容越完整、面向越多，分數越高。

進度檢查要求：

- 不需要提前寄送投影片給助教。
- 需自備筆電。
- 需確認電腦可支援 VGA 或 HDMI 連接。
- 評分依據是分享內容的完整性與多樣性。

另外，期末最後一週會有選擇性口頭報告，可提供最多 5% 額外加分。但名額受限，若報名組數過多，會優先考慮較早報名的組別。助教也警告，這 5% 不是保證給滿；若模型表現很差，例如連 baseline 都未超過，口頭報告不會自動拿滿分。

### 6. 學術誠信與一致性：最容易被低估的零分風險

課程團隊對違規風險說得非常直接：

- 不可抄襲他人程式碼。
- 不可直接使用他人分數或結果。
- 報告、程式、sample result 必須一致。
- 若報告描述與實際程式不符，相關部分可能給零分。
- 可使用 AI 工具輔助，但必須檢查生成內容。
- 若報告含有 AI 幻覺、未查證引用或直接整篇生成未校對，報告會被大幅扣分。
- 若助教發現提交有疑慮，可能會聯絡學生確認情況。

這是課程團隊在預先劃紅線。真正的要求不是「不能用工具」，而是「不能失去責任鏈」。模型、程式、報告、結果必須能互相對上。
